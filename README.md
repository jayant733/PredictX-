# Nexus Prediction Market Monorepo

Nexus is a high-performance prediction market platform built with a Turborepo + Bun monorepo. It features Solana wallet Web3 authentication, orderbook-based trading with cross-market (reverse orderbook) matching, capital splits/merges, and a Solana deposit indexer.

---

## 🏗️ Repository Architecture

```
prediction-market/
├── apps/
│   ├── api/                 # REST API server (Bun.serve) exposing Web3 Auth and Trade endpoints
│   └── indexer/             # Solana deposit listener daemon (polling and WebSocket onAccountChange)
├── packages/
│   ├── db/                  # Drizzle ORM schemas, migration config, and connection pools
│   └── orderbook/           # Core matching engine with splits/merges and PostgreSQL locking
└── docs/                    # Frontend UI specifications for builder/stitch
    ├── login.md             # Wallet connect & Supabase custom JWT flow
    ├── dashboard.md         # Active markets, filters, and user balances
    ├── market-detail.md     # Limit/Market forms, orderbooks, and interactive charts
    └── portfolio.md         # Unrealized P&L, holdings table, and transaction history
```

## ⚙️ Setup & Environment Variables

Create a `.env` file at the root or inject variables into your environment:

```env
# Database connection string (Postgres/Supabase)
DATABASE_URL="postgres://postgres:postgres@localhost:5432/nexus"

# Must match Supabase JWT secret to allow Supabase to verify JWTs generated by the API
JWT_SECRET="your-supabase-jwt-secret-minimum-32-chars-long"

# AES-256-GCM secret for encrypting generated deposit addresses' private keys in the database
ENCRYPTION_SECRET="your-encryption-key-for-deposit-wallets"

# Solana RPC connection
SOLANA_RPC_URL="https://api.devnet.solana.com"

# Fixed exchange rate in development (1 SOL = $150.00)
SOL_PRICE_USD_CENTS=15000

# Port for the REST API
PORT=3001
```

## 🚀 Getting Started

### 1. Install Dependencies
```bash
bun install
```

### 2. Set Up Database Schema
Apply the Drizzle schemas and create the database tables:
```bash
cd packages/db
bun run db:push
```

### 3. Run Development Servers
Start the API and Solana deposit indexer simultaneously using Turborepo:
```bash
bun run dev
```

## 📈 Orderbook Cross-Matching Rules

To maximize liquidity, the matching engine supports **Direct Matches** and **Cross-Market Matches** (also known as reverse orderbook matching):

* **Direct Buy Match**: `BUY YES` matches resting `SELL YES` at or below limit price.
* **Cross Buy Match**: `BUY YES` matches resting `BUY NO` if $Price(YES) + Price(NO) \ge 100$. The system takes the capital from both buyers, splits it to mint 1 YES and 1 NO share, and distributes them.
* **Direct Sell Match**: `SELL YES` matches resting `BUY YES` at or above limit price.
* **Cross Sell Match**: `SELL YES` matches resting `SELL NO` if $Price(YES) + Price(NO) \le 100$. The system takes 1 YES and 1 NO share from the sellers, merges them into $1.00 cash, and pays both sellers.

---

## 🛡️ Concurrency & Database Locking

All order placement, matches, splits, and merges are executed inside PostgreSQL database transactions.
* **Deterministic Lock Ordering**: To prevent deadlocks, the engine locks the target `market` row using `SELECT ... FOR UPDATE` before performing any operations on orderbooks or balances. This serializes matching queue updates per market.
* **Double Spend Protection**: The user's account row is locked via `SELECT balance_usd FROM users ... FOR UPDATE` before updating balances.
